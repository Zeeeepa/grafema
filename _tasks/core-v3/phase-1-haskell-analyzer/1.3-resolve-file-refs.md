# Task 1.3: Implement resolveFileRefs (Post-File Scope Resolution)

## Goal

After `walkProgram` produces the full `FileAnalysis`, a `resolveFileRefs` post-pass walks the scope chain for each `ScopeLookup`/`CallResolve` deferred ref. Resolved refs → emit edge. Unresolved → REFERENCE node with `resolved: false` metadata.

## Key Design Decision

**Option C — derive scope tree from emitted graph data.** The scope tree is reconstructed from SCOPE nodes, HAS_SCOPE edges, and DECLARES edges (populated by task 1.1). No monad changes needed — pure post-processing of `FileAnalysis` output.

## Files to Create/Modify

- **NEW**: `Analysis/Resolve.hs`
- **Modify**: `Analysis/Context.hs` (wire resolveFileRefs into pipeline)
- **Modify**: `Main.hs` (pipe walkProgram output through resolveFileRefs)

## Subtasks

### 1.3a: Create Analysis/Resolve.hs
Define `resolveFileRefs :: FileAnalysis -> FileAnalysis` as the main entry point. Pure function — takes analysis output, returns modified output with resolved edges.

### 1.3b: Build scope tree index
From emitted graph data, construct `Map ScopeId (Map Name Declaration)`:
- SCOPE nodes define scopes with their parent scope IDs
- DECLARES edges (from declareInScope in 1.1) link scopes to declarations
- HAS_SCOPE edges define the parent-child scope tree
- Result: an indexed scope tree for O(1) lookup at each scope level

### 1.3c: resolveInScope
Implement `resolveInScope :: ScopeId -> Name -> ScopeTree -> Maybe Declaration`
- Start at the given scope, look up name
- If not found, walk to parent scope
- Repeat until found or root scope reached
- Return the Declaration if found, Nothing otherwise

### 1.3d: Resolved refs → emit edges
For each resolved DeferredRef:
- Create edge: `source = drFromNodeId`, `target = declNodeId`, `type = drEdgeType`
- Edge type comes from the DeferredRef kind (ScopeLookup → READS_FROM, CallResolve → CALLS, etc.)
- Add resolved edges to `faEdges`

### 1.3e: Unresolved ScopeLookup/CallResolve → REFERENCE(resolved:false)
For deferred refs that don't resolve in any scope:
- Leave the existing REFERENCE node in place
- Add/update metadata: `resolved: false`
- These will be resolved later by cross-file plugins (Phase 3)

### 1.3f: ImportResolve/ExportLookup passthrough
Import and export deferred refs are cross-file by nature:
- Pass through as IMPORT_BINDING nodes unchanged
- They'll be resolved by the import resolution plugin (task 3.3)
- No scope chain walk needed for these

### 1.3g: Wire into Main.hs
Pipeline: `runAnalyzer ... (walkProgram program)` → pipe result through `resolveFileRefs`
```haskell
let rawAnalysis = runAnalyzer env (walkProgram program)
let resolved = resolveFileRefs rawAnalysis
-- encode resolved for output
```

### 1.3h: Handle var hoisting
`var` declarations are visible throughout their function scope, not just after the declaration line. The scope tree index must account for this:
- `var` → registered at function scope level (even if declared in a block)
- `let`/`const` → registered at block scope level (already handled by 1.1a)
- Function declarations → hoisted to enclosing function scope

## Acceptance Criteria

- Intra-file references like `const x = 1; f(x)` produce a READS_FROM edge from CALL to VARIABLE
- Cross-file references remain as IMPORT_BINDING or REFERENCE(resolved:false) nodes
- Var hoisting works: `function f() { console.log(x); var x = 1; }` — x resolves to the var declaration
- Shadowing works: inner scope declaration shadows outer
- Closure references work: inner function referencing outer function's variable

## Dependencies

- **1.2** (drScopeId must be populated for scope chain walk start)
- **1.5** (ExportInfo needed to distinguish export-related refs)
