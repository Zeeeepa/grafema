# Task 2.7: Generation-Based Garbage Collection

## Goal

Bump generation counter per analysis run. Stamp all nodes/edges with generation. After completion, delete stale data from changed files.

## Files to Modify

- `packages/grafema-orchestrator/src/gc.rs`

## Subtasks

### 2.7a: Generation counter management
- Read current generation from RFDB metadata (or start at 1)
- Increment at start of each analysis run
- Persist new generation number in RFDB metadata node

### 2.7b: Stamp _generation on all emitted data
All nodes and edges emitted by the analyzer or plugins get `_generation: <current>` metadata:
- Analyzer output: stamp during ingestion (task 2.5c)
- Plugin output: stamp during validation (task 2.6e)

### 2.7c: Delete stale data after CommitBatch
After analysis completes for changed files:
```rust

    // For each changed file: delete nodes/edges where
    //   file == changed_file AND _generation < current_gen
    // This removes nodes from previous analysis that are no longer emitted
}
```
- Use `changedFiles` in CommitBatch to scope GC to modified files only
- Unchanged files keep their existing nodes (no re-analysis needed)

### 2.7d: Incremental analysis
Only re-analyze files with mtime > last_analyzed:
- Track `last_analyzed_mtime` per file in RFDB metadata
- Compare with current file mtime
- Skip files that haven't changed
- Force re-analysis with `--force` flag

## Acceptance Criteria

- Re-running analysis on unchanged files is idempotent (no duplicate nodes)
- Changed files get new nodes, old nodes cleaned up
- Deleted files: all their nodes/edges removed
- New files: analyzed and ingested normally
- `--force` flag re-analyzes all files regardless of mtime

## Dependencies

- **2.2** (RFDB client for metadata storage and deletion)

## Estimated Scope

~0.5 day. Generation stamping is mechanical; mtime tracking is the main logic.
