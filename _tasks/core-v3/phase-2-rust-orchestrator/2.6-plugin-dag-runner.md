# Task 2.6: Plugin DAG Runner

## Goal

Execute plugins in DAG order based on `depends_on`. Support streaming and batch modes. This is the core extensibility mechanism of core-v3.

## Files to Modify

- `packages/grafema-orchestrator/src/plugin.rs`

## Reference

- Core-v3 plan constraints: C1 (validate before write), O4 (per-plugin timeout), R3 (stamp metadata)

## Subtasks

```rust
    // Returns levels: [[independent plugins], [depends on level 0], ...]
    // Error if cycle detected

```
- Validate: no unknown `depends_on` references
- Detect cycles (error with clear message showing the cycle)

### 2.6b: Streaming mode execution
For plugins with `mode: Streaming`:
1. Execute plugin's `query` against RFDB → get matching nodes
2. Serialize nodes as NDJSON (newline-delimited JSON) to plugin stdin
3. Read plugin stdout: `emit_node` / `emit_edge` commands (NDJSON)
4. Buffer output for validation before writing to RFDB

### 2.6c: Batch mode execution
For plugins with `mode: Batch`:
1. Set `RFDB_SOCKET` environment variable
2. Set `RFDB_DATABASE` environment variable
3. Spawn plugin command
4. Plugin queries RFDB directly, writes results directly
5. Wait for completion (with timeout)

### 2.6d: Output validation (C1)
Before writing streaming plugin output to RFDB:
- Validate node IDs are well-formed
- Validate edge source/target reference existing nodes
- Validate required metadata fields present
- Reject malformed output with clear error (don't write partial results)

### 2.6e: Stamp metadata (R3)
On all plugin-emitted nodes/edges, add:
- `_source: "<plugin_name>"` — which plugin created this
- `_generation: <current_generation>` — for GC (see task 2.7)

### 2.6f: Per-plugin timeout (O4)
- Default timeout from config (e.g., 60s)
- Per-plugin override via `timeout` field
- On timeout: kill process, emit ISSUE node with plugin name and timeout duration
- Don't block other plugins in the same DAG level

### 2.6g: Parallel execution of independent plugins
Plugins at the same DAG level (no dependencies between them) run in parallel:
```rust
for level in dag_levels {
    let handles: Vec<_> = level.iter().map(|plugin| {
        tokio::spawn(run_plugin(plugin))
    }).collect();
    join_all(handles).await;
}
```

## Acceptance Criteria

- Given 2 plugins where B depends on A: A runs first, B runs after A completes
- Independent plugins at same level run in parallel
- Plugin output appears in RFDB with `_source` and `_generation` metadata
- Malformed plugin output is rejected (not partially written)
- Timeout kills plugin and emits ISSUE node
- Cycle in plugin DAG produces clear error at startup (not at runtime)
